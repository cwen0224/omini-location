# 人權博物館APP 接手工作流與踩坑紀錄

## 0. MerFork Protocol
這份文件記錄的整條更新與發版鏈，正式名稱為 `MerFork Protocol`。

它涵蓋的不是單一腳本，而是以下完整流程：
- 本機 build release APK
- 同步 `pubspec.yaml`、`version.json`、`index.html`
- 上傳 GitHub Releases
- 由 GitHub Pages 提供下載頁與回報查看頁
- 由 Supabase 儲存問題回報、Beacon、測試資料

如果接手者只記得一個名字，請先記 `MerFork Protocol`，再去看下面各段細節。

## 1. 文件目的
這份文件不是需求規格，也不是產品提案。  
它的用途是讓下一位 AI 或工程師接手時，直接照著目前已驗證可行的工作流操作，不必再重建上下文。

適用對象：
- AI 代理
- 新接手工程師
- 需要臨時協助維護專案的人

最後驗證狀態日期：
- 2026-03-13

- `allow anon select app_errors`
- `allow anon insert error_attachments`
- `allow anon select error_attachments`

## 7. 已踩過的關鍵坑

### 7.1 手機抓不到新版本，不一定是 App 壞了
最常見原因不是更新機制壞掉，而是：

- `app/pubspec.yaml`
- `docs/version.json`
- `docs/index.html`

改了但還沒 `commit / push`。

實務上曾發生：
- 本機已 build 出新 APK
- GitHub Release asset 也已上傳
- 但 `version.json` 還停留在本機未 push
- 手機因此仍判斷沒有新版

結論：
- 手機看的是遠端 `version.json`
- 不是你本機的 `pubspec.yaml`

### 7.2 `publish_github_release.bat` 原本會「看起來成功，其實鏈沒閉合」
已修正。

現在這支腳本會在以下情況直接報錯：
- `app/pubspec.yaml` / `docs/version.json` / `docs/index.html` 有未提交變更
- 上述檔案有 staged 但未 commit
- 本地 branch 尚未 push 到 upstream

目的：
- 避免又出現「GitHub Release 有了，但手機抓不到更新」的假成功

### 7.3 `build_release.bat` 曾經因為黑箱輸出讓人以為卡死
已修正。

現在 build script 會顯示：
- Step 1 停止 Gradle daemon
- Step 2 清理 dex
- Step 3 清理 APK outputs
- Step 4 Flutter doctor
- Step 5 Flutter pub get
- Step 6 Flutter build apk

完整細節仍寫入：
- `logs/build_release.log`

### 7.4 批次檔改壞時，先查 `logs/build_release.log`
不要先猜手機或 Flutter 壞掉。

曾經發生的真實例子：
- 批次檔 quoting 壞掉
- Step 1 直接報：
  - `檔案名稱、目錄名稱或磁碟區標籤語法錯誤`

後來修正方式：
- Windows `cmd /c` 內部引號要用雙引號包法
- Flutter 那幾步不要把步驟名稱一起當命令參數送進去

### 7.5 截圖回報在 release 模式曾經失敗
真實錯誤：
- `LateInitializationError`
- 來源是 `RenderRepaintBoundary.debugNeedsPaint`

原因：
- 這種 debug-only 判斷在 release 模式踩雷

修法：
- 改成等待 `endOfFrame`
- 再做少量重試
- 不再依賴 `debugNeedsPaint`

對應檔案：
- `app/lib/core/app_capture_service.dart`

### 7.6 Render 免費版不適合當正式報錯收件箱
這條路實際評估後放棄，原因：

- 免費版不適合持久存檔
- 服務會休眠
- 若要存截圖與歷史回報，不夠穩

最後定案：
- 仍用 Supabase 當資料來源
- GitHub Pages 提供查看頁

## 8. 正確發版順序
這是目前已驗證可行的正式順序：

1. 修改 `app/pubspec.yaml`
2. 修改 `docs/version.json`
3. 修改 `docs/index.html`
4. `git add / commit / push`
5. 執行 `publish_github_release.bat`
6. 確認 GitHub Release asset 存在
7. 等 GitHub Pages 更新
8. 手機重新檢查更新

不要反過來。

錯誤順序：
- 先跑 `publish_github_release.bat`
- 再去 commit / push 版本檔

這樣很容易造成：
- GitHub Release 已有新 APK
- 手機仍只看到舊版 manifest

## 9. 問題排查順序
遇到問題時，照這個順序排：

### 9.1 手機抓不到更新
先查：

1. `docs/version.json` 遠端是不是新版本
2. `apk_url` 是否指到對應 tag
3. GitHub Release asset 是否存在
4. 手機當前安裝版本是否真的比較舊

### 9.2 回報按鈕沒反應
先查：

1. 是否真的有進 `回報問題` 頁
2. `ErrorReporter` 最近事件
3. `report-viewer.html` 是否有收到最新回報

### 9.3 截圖沒上傳
先查：

1. 回報頁有沒有截圖預覽
2. `report-viewer.html` 那筆回報的 `context_json`
3. 是否有：
   - `screenshot_path`
   - `screenshot_url`

### 9.4 `.bat` 看起來卡住
先查：

1. `logs/build_release.log`
2. 是否已進入 `assembleRelease`
3. `java` / `dartvm` 是否仍在跑

不要先判斷成「閃退」。
