## 2. 專案目標
建立一套可在室內外運作的手機定位系統，混合使用以下四種來源，降低室內 GPS 訊號不佳問題:
- GPS
- 相機 (Visual / VIO / SLAM)
- BLE Beacon
- 陀螺儀/IMU

## 3. 目前已確認需求
- 必須支援多感測器融合定位，不採單一 GPS。
- 目標場域包含室內空間，且需處理 GPS 弱訊號或不可用情境。
- 技術方向以開源方案優先。
- 希望方案可實作、可持續擴充與維護。
- 系統需支援兩種可切換模式:
  - 開發者模式: 現場蒐資、建構虛擬地圖與資料集。
  - 使用者模式: 依既有地圖定位並留下訊息紀錄。
- 系統需支援多數手機，採分級能力策略，避免高階功能成為必要門檻。
- 系統需提供地圖 AR 指引，協助使用者依現場視角理解行進方向。
- 系統目標為提升複雜場域的定位穩定性與空間辨識能力，並逐步強化沉浸式導覽體驗。
- App 需支援首次安裝後的版本更新提示與更新包下載，降低每次手動重裝 APK 的成本。
- App 需支援內容更新包機制，使地圖、展點、Beacon 設定與文案可在不重裝 APK 的前提下更新。

## 4. 定位策略 (高層)
- 戶外主定位: GPS 提供絕對座標。
- 室內主定位: BLE + 相機 + IMU。
- 過渡區域 (室內外出入口): 動態調整感測器權重，避免定位跳點。
- 漂移校正: 使用 BLE 與可用 GPS 作為絕對參考，修正視覺/IMU 累積誤差。

## 5. 融合演算法需求
- 可行做法:
  - EKF/ESKF (即時性高，MVP 優先)
  - 因子圖優化 (精度提升，後續迭代)
- 必要能力:
  - 感測器時間同步
  - 量測可信度加權 (GPS、BLE、相機品質分數)
  - 離群值剔除與失效保護

### 5.1 權重決策規則 (GPS/相機/BLE/IMU)
- 權重採動態計算，不使用固定比例。
- 每個模組先計算即時品質分數 `q` (0~1):
  - GPS: HDOP、衛星數、訊號強度、更新延遲
  - 相機: 追蹤狀態、特徵點數、重投影誤差、影像清晰度
  - BLE: 可見 beacon 數量、RSSI 波動、空間幾何分布
  - IMU: 偏移估計穩定度、飽和狀態、時間同步品質
- 依場域套用情境係數 `c`:
  - 室內: 降低 GPS，提升相機與 BLE
  - 室外: 提升 GPS，降低 BLE
  - 過渡區: 3~5 秒平滑切換，避免跳點
- 最終權重計算:
  - `w_i = normalize(q_i * c_i)`
  - EKF 量測噪聲建議: `R_i = R_base_i / (w_i + e)`
- 衝突處理:
  - 使用馬氏距離/卡方門檻做 gating，離群量測拒收或降權
  - 若多模組分歧過大，輸出低可信度並觸發重定位
- 初始基準權重 (可調):
  - 室外: GPS 0.55 / 相機 0.20 / BLE 0.10 / IMU 0.15
  - 室內: GPS 0.05 / 相機 0.45 / BLE 0.35 / IMU 0.15

## 6. 模式需求
### 6.1 開發者模式 (Developer Mode)
- 可在現場啟動資料蒐集任務。
- 蒐集資料來源:
  - 相機影像/特徵
  - IMU 序列
  - BLE Beacon 掃描結果 (UUID/Major/Minor/RSSI)
  - GPS (若可用)
- 可建立或更新虛擬地圖 (Map Dataset)。
- 可輸出建圖成果與版本資訊，供使用者模式載入。
- 需保留建圖過程紀錄與品質指標 (覆蓋率、可追蹤區段、時間戳)。

### 6.2 使用者模式 (User Mode)
- 載入指定版本地圖資料。
- 即時估計手機在地圖中的位置與朝向。
- 當定位可信度低時，提供重定位機制。
- 支援使用者在當前位置留下訊息紀錄。
- 訊息紀錄需綁定定位資訊與時間戳。
- 提供地圖 AR 指引 (箭頭、路徑或目標標記) 以輔助導覽。
- App 啟動時需可檢查是否有新版安裝包或內容更新包。

### 6.3 模式切換
- 需可在 App 內明確切換模式，避免混淆。
- 使用者模式預設不可修改地圖資料。
- 開發者模式可建立新地圖版本，但需留下操作紀錄。

## 7. 功能需求 (第一版)
- 手機端即時定位輸出 (座標、方向、置信度)
- 感測器狀態監控 (GPS 品質、BLE 訊號、相機追蹤狀態、IMU 健康度)
- 室內外切換邏輯
- 追蹤中斷後重定位能力
- 地圖資料載入、切換與版本控管
- 位置綁定訊息新增與查詢
- 裝置能力檢測 (是否支援相機增強定位)
- 相機定位失效時自動退化為 BLE + IMU + GPS
- 支援動態權重計算、離群值拒收與低可信度警示
- AR 指引畫面與一般地圖畫面可切換
- AR 指引需顯示目前位置到目標點的方向提示
- AR 指引需顯示定位可信度不足的警示狀態
- App 啟動時需支援版本檢查
- 若有新版 APK，需顯示更新提示、版本資訊與下載入口
- 若有新版內容包，需支援背景下載、版本比對與本地套用
- App 需提供目前安裝版本、內容版本與最後更新時間的資訊頁

## 8. 資料需求 (第一版)

## 10. 技術選型 v1 (主選/備選/授權)
### 10.1 感測器與定位融合
- 主選:
  - OpenCV: 影像前處理、特徵抽取、PnP/姿態估計。
  - RTAB-Map: 開發者模式建圖 + 使用者模式視覺定位。
  - Ceres Solver: 多來源誤差最小化與軌跡優化。
- 備選:
  - ORB-SLAM3: 高精度 SLAM 備選引擎。
  - OpenVINS: 視覺慣性估計備選。
- 授權重點:
  - OpenCV (Apache-2.0)、RTAB-Map (BSD-3-Clause)、Ceres (BSD-3-Clause) 可商用。
  - ORB-SLAM3 (GPL-3.0) 具傳染性授權風險，預設列為備選。

### 10.2 BLE 與近距定位
- 主選:
  - AltBeacon Android Beacon Library: Android 端掃描/測距。
- 備選:
  - OpenHPS BLE 模組: 做跨平台資料流整合參考。
- 授權重點:
  - AltBeacon Library (Apache-2.0)；AltBeacon 規格可自由實作。

### 10.3 地圖與空間資料
- 主選:
  - MapLibre Native / MapLibre GL JS: 手機與管理端地圖渲染。
  - OpenStreetMap: 基礎地圖資料。
  - PostgreSQL + PostGIS: 空間資料庫與查詢引擎。
  - QGIS: 場域圖資編修與 QA。
- 備選:
  - CesiumJS: 若後續需要更強 3D 地圖能力。
- 授權重點:
  - OSM 使用 ODbL，衍生資料需遵守授權條件。

### 10.4 AR 指引層
- 主選:
  - three.js + A-Frame + AR.js: AR 路徑與目標標記呈現。
  - AprilTag: 現場錨點校正與快速重定位。
- 備選:
  - 以原生 ARKit/ARCore 路線做第二階段效能強化。
- 授權重點:
  - three.js / A-Frame / AR.js / AprilTag 皆可商用開源授權。

### 10.5 選型結論 (v1)
- MVP 預設組合:
  - 定位核心: OpenCV + RTAB-Map + Ceres
  - BLE: AltBeacon (Android) + OpenHPS 架構參考
  - 地圖資料: MapLibre + OSM + PostGIS + QGIS
  - AR 指引: three.js + A-Frame + AR.js + AprilTag
- v1 狀態: 候選完成，待 POC 驗證後升級為 v2 定案。

  - 評估原則: 是否導入需依館內實地測試結果判定
  - 主要考量: 深層室內收訊不穩、牢房級精度需求、文化影像資料主權

## 11. 風險與限制
- 室內多路徑與遮蔽會影響 GPS/BLE 穩定性。
- 視覺定位受光照、動態人流、低紋理區影響。
- 不同手機型號感測器品質差異大。
- 授權相容性需在商用前完成審查。
- 地圖版本更新後，舊版訊息與座標對齊需定義相容策略。
- 行動作業系統版本與權限政策變動可能影響定位穩定性。
- AR 指引可能受相機漂移、遮擋與低光環境影響體驗。
- 更新包若缺乏版本控管與 checksum，可能造成內容版本錯置或無法回退。
- Android 側載更新仍受系統安全限制，需設計清楚的引導流程。

## 15. 館方提案文稿 (修訂版)
國立人權博物館：智慧導覽與數位空間定位系統提案

本計畫依據人權博物館（不義遺址）之場域特性，規劃整合 GPS、相機辨識（CV）、藍牙（BLE）與慣性測量元件（IMU）等技術，建置一套適用於複雜室內外環境之語音導覽定位系統。  
系統目標在於改善既有導覽於特定場域中的定位穩定性與空間辨識能力，並在兼顧設備條件與維運成本的前提下，逐步提升參觀體驗品質。

### 15.1 為什麼 Google Geospatial（LBS）未必適合直接作為本館主要方案？
在人權博物館此類場域中，傳統雲端 LBS（如 Google Geospatial）在特定使用情境下可能面臨以下限制；是否適用仍需依實地測試結果判定：
1. 建築結構可能影響訊號穩定性
  - 園區內如押解室、牢房、法庭等空間，多為厚重鋼筋混凝土建築，GPS 訊號及行動網路可能因遮蔽而不穩，進而影響雲端定位服務的可用性與一致性。
2. 室內空間辨識精度可能不足
  - 牢房空間較小、相鄰點位密集，若僅依賴傳統 LBS 常見誤差範圍（約 5-10 公尺），在部分區域可能較難穩定判斷使用者所在位置，增加導覽內容誤觸或錯置的風險。
3. 場域資料管理與敏感性考量
  - 園區影像與空間資訊具有歷史敏感性與保存價值。若採用私有化架構，可降低環境影像特徵資料送往第三方雲端的需求，並有助於館方在資料管理與權限控管上保有較高彈性。

### 15.2 分級執行方案
- 互動型方案: 文物精準導引
  - 技術核心: 藍牙定位 + 主動式視覺校準（Image Tracking）
  - 定位方式: 在自動語音觸發之外，針對特定史料、書信、牆面文字或文物，導引使用者以相機對準目標（或掃描 QR Code），進行位置與物件辨識校準。
  - 導覽體驗: 在適當校準條件下，可較準確對應使用者所面向之牢房、展件或史料，觸發與該物件相關的解說內容，以降低相鄰點位誤觸的情況。
  - 方案特點:
    - 較適合展示點位密集的區域
    - 可作為改善重疊觸發問題的實務做法之一
    - 後續可評估延伸互動式教學應用（如史料探索、導覽任務）
  - 定位精度（預估）: 在標記設計、光線條件與校準流程配合下，可達約 1 公尺內（物件級觸發）；實際表現仍需以場域測試為準。
- 沉浸型方案: AR 歷史再現
  - 技術核心: 私有化空間點雲（VIO SLAM）+ 3D 空間音訊
  - 定位方式: 先建立園區數位空間地圖（Digital Twin），在完成場域建圖與裝置校準後，系統可於離線條件下維持 AR 空間定位與內容對位能力。
  - 導覽體驗: 可結合 AR 視覺與空間音訊，讓使用者在現場觀看歷史場景的疊加呈現，並透過聲音方位設計提升情境理解（例如法庭、監所等情境之聲音來源方向）。
  - 方案特點:
    - 可作為較高沉浸度之展示形式選項
    - 在特定區域可降低對即時連網的依賴
    - 較適合作為特展、教育推廣或示範場域之進階版本
  - 定位精度（預估）: 視場域建圖品質、裝置性能、光線環境與追蹤條件而定；可達較高精度，但仍需以實機測試結果評估。
- 普及型方案: 沉浸式口袋導覽
  - 技術核心: 低功耗藍牙（BLE）+ 慣性導航（PDR）
  - 定位方式: 於展間適當位置設置 Beacon，系統依訊號強度與移動狀態推估使用者所處區域。
  - 導覽體驗: 以區域自動觸發為主。使用者配戴耳機、手機放入口袋，可在進入特定展區（如醫務室、福利社）時播放對應語音內容。
  - 方案特點:
    - 對手機設備規格要求相對較低
    - 參觀過程可減少操作手機頻率
    - 較有機會維持場域參觀的專注與安靜氛圍
  - 定位精度（預估）: 約 3-5 公尺（區域級觸發），實際表現會受 Beacon 佈點密度、場域干擾與人流影響。

### 15.3 方案價值與開銷分析（初步比較）
以下比較為初步規劃階段之方向性評估，實際成本、效能與限制仍需依場域測試、設備規格、授權方案與維運範圍進一步確認。

| 項目 | 本專案建議（離線私有化） | 第三方 LBS（Google Geospatial） |
|---|---|---|
| 環境適應力 | 相對較佳（可依場域特性調整，較適合室內遮蔽或收訊不穩區域） | 受 GPS/網路條件影響較大，在部分區域可能較不穩定 |
| 資安與隱私 | 館方可保有較高資料控管彈性（依實際部署架構而定） | 部分功能可能需依賴第三方雲端服務與其資料流程 |
| 長期開銷 | 前期建置投入可能較高；後續以維護、更新與場域調整為主 | 可能涉及服務授權或使用量計費，成本結構需依方案確認 |
| 硬體相容性 | 可依方案分級支援不同手機條件（A/B 方案相對較廣） | 部分 AR 功能可能較仰賴新款裝置規格 |
