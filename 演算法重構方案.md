# 人權博物館APP 演算法重構方案

## 1. 文件目的
本文件用來取代目前過度依賴 heuristic 與頁面級測試邏輯的定位做法，定義後續正式演算法應採用的核心方向。

這份文件要解決的問題是：
- 目前的相對移動估計太粗糙
- BLE、IMU、羅盤、Camera、GPS 之間沒有統一的融合框架
- 室內外切換還停留在功能驗證層，不是正式的 handover 設計
- Camera 與 BLE 尚未被明確定義為「修正源」或「區域觀測源」

## 2. 目前做法的問題
### 2.1 不應再延續的做法
- 不要把 IMU 加速度直接粗略積分後當成正式位置
- 不要把 BLE RSSI 直接硬轉距離再當成位置真值
- 不要把羅盤 heading 當成永遠可信的朝向
- 不要把 Camera 一啟動就直接當主定位來源
- 不要用頁面級 heuristic 當正式定位核心

### 2.2 目前版本的正確定位
目前 App 內已有的：
- GPS 軌跡顯示
- IMU / 羅盤相對位移視覺化
- BLE / Camera 測試頁
- 全模組測試頁

這些內容適合：
- 資料蒐集
- 感測器驗證
- 操作流程打通
- 參數觀察

但不適合直接當作正式定位演算法。

## 3. 正式演算法的總體結論
後續正式定位核心建議採用：

`IMU 預測 + 動態權重 EKF / Error-State KF + BLE 區域修正 + Camera 重定位 / 強修正 + GPS 室外絕對基準 + 室內外 handover`

這代表：
- IMU 是高頻主幹，不是最後真值
- 羅盤是 heading 約束，不是單獨位置來源
- BLE 是區域觀測或距離約束，不是單獨真值
- Camera 是姿態/位置修正或重定位來源
- GPS 是室外絕對位置基準
- 室內外切換必須是一個正式狀態，不是單純 if/else

## 4. 五模組的角色分工
### 4.1 GPS
角色：
- 室外絕對位置基準
- 室外速度參考
- 過渡區弱約束

應用原則：
- 室外時高權重
- 室內時大幅降權
- 過渡區平滑切換

### 4.2 IMU
角色：
- 高頻運動預測
- 短時間連續位姿估計
- 定位主幹的 prediction model

應用原則：
- 一直運作
- 但必須被外部觀測源修正
- 不可長時間單獨放飛

### 4.3 羅盤儀
角色：
- heading 約束
- 轉向估計修正
- 協助 BLE / Camera 解釋朝向

應用原則：
- 要有磁干擾品質分數
- 不可信時要降權
- 只能當 orientation constraint，不該單獨決定位置

### 4.4 BLE Beacon
角色：
- 室內區域觀測
- Beacon 區域識別
- 相對距離或位置修正

應用原則：
- 優先做 fingerprint / 區域判定
- 其次才做距離約束
- 不直接把 RSSI 轉成真實座標

### 4.5 Camera
角色：
- 視覺重定位
- 姿態修正
- 特定條件下的高可信絕對修正

應用原則：
- 平常不必永遠當主定位
- 在可追蹤、特徵足夠、品質高時做 correction
- 失效時要快速退回 IMU + BLE / GPS

## 5. 核心融合架構
### 5.1 濾波器
第一版建議：
- EKF 或 Error-State Kalman Filter

原因：
- 適合手機即時運算
- 能統一處理 prediction 與 measurement update
- 適合逐步擴充，不需要一開始就上因子圖

第二階段可考慮：
- 因子圖優化
- smoothing / replay analysis

### 5.2 狀態向量
第一版建議狀態：
- `x`
- `y`
- `z`
- `vx`
- `vy`
- `vz`
- `yaw`
- `imu_bias_acc`
- `imu_bias_gyro`

若第二階段要更完整：
- `roll`
- `pitch`
- `camera_pose_confidence`
- `ble_zone_confidence`

### 5.3 prediction
prediction 主要由 IMU 提供：
- 加速度
- 角速度
- 偏移補償
- 時間戳積分

### 5.4 update
update 來源：
- GPS measurement update
- BLE measurement update
- compass heading update
- camera pose update

## 6. 動態權重與品質分數
每個模組都必須有 quality score。

### 6.1 GPS quality
- HDOP / accuracy
- 衛星數
- 更新延遲
- 速度合理性

### 6.2 IMU quality
- 偏移穩定度
- 飽和狀態
- 取樣穩定度
- 積分漂移估計

### 6.3 羅盤 quality
- 地磁強度是否異常
- heading 抖動程度
- 與 gyro 推定轉向是否矛盾

### 6.4 BLE quality
- 可見 Beacon 數量
- RSSI 波動
- 空間幾何分布
- 是否進入已知 Beacon 區域

### 6.5 Camera quality
- tracking state
- feature score
- relocalization success
- 畫面穩定度

### 6.6 權重原則
即時權重不是寫死，而是：

`w_i = normalize(q_i * c_i * mode_i)`

其中：
- `q_i` = 模組品質
- `c_i` = 場域情境係數
- `mode_i` = 目前模式或 handover 狀態

## 7. 室內外 handover
### 7.1 不能再用單純切換
室內外切換不是：
- 室外用 GPS
- 室內用 BLE

而是要有一個中間狀態。

### 7.2 建議狀態
- `outdoor`
- `transition`
- `indoor`

### 7.3 規則
`outdoor`
- GPS 高權重
- Camera/BLE 為輔助

`transition`
- GPS 權重逐步下降
- BLE / Camera 逐步上升
- 3~5 秒平滑

`indoor`
- GPS 幾乎不主導
- BLE + Camera + IMU 為主

## 8. BLE 的正確策略
### 8.1 第一版建議
- 先做 Beacon fingerprint / zone classification
- 再把 zone 或少數高可信 Beacon 當 measurement update

### 8.2 不建議
- 不要單靠 RSSI -> distance 直接解位置
- 不要把浮動 remoteId 當最終 Beacon 身分

### 8.3 App 資料要求
必須保留：
- beacon_key
- manufacturer data
- service data
- saved display name
- 該 Beacon 所屬區域

## 9. Camera 的正確策略
### 9.1 第一版
- Camera 提供 relocalization / pose correction
- 不要求永遠持續高頻定位

### 9.2 條件式啟用
只有在以下條件好時才高權重：
- tracking state 良好
- feature score 足夠
- 畫面穩定
- 已知地標或視覺地圖可用

### 9.3 失效退化
失效時回到：
- IMU 主幹
- BLE 修正
- GPS 若可用則輔助

## 10. 羅盤與 heading 的正確策略
官方與實務都顯示 yaw/heading 不是永遠可靠。

因此：
- heading 必須帶 accuracy / confidence
- 羅盤應與 gyro 轉向一致性檢查
- 磁干擾高時要降權

## 11. 實作分期
### Phase A：資料化
先把目前 guided session 的資料流補齊：
- sensor_samples
- user_feedback
- ground_truth_points
- derived_metrics

### Phase B：品質分數
離線分析：
- GPS quality
- BLE quality
- compass quality
- camera quality
- imu drift score

### Phase C：融合器骨架
實作：
- state vector
- prediction
- measurement update
- covariance update

### Phase D：handover
加上：
- indoor / transition / outdoor 狀態機
- 平滑切換
- 模組降權與升權

### Phase E：重定位與回放
加上：
- session replay
- error analysis
- tuning dashboard

## 12. 對目前專案的直接要求
### 12.1 可以保留的
- 感測器測試頁
- 全模組測試頁
- 引導式定位建檔
- Supabase 資料上傳

### 12.2 必須逐步淘汰的
- 把相對移動圖誤當定位真值
- 頁面內 heuristic 當正式演算法
- BLE / 羅盤 / Camera 個別直出位置，不經融合器

### 12.3 接下來應優先做
1. 在 Supabase 補 `quality metrics`
2. 新增 `LocalizationState` 與 `Measurement` model
3. 做第一版 EKF skeleton
4. 建 `handover state machine`
5. 建 `session replay / tuning` 工具

## 14. 結論
後續正式定位的關鍵不是再堆更多測試頁，而是：
- 把五模組的角色釐清
- 用統一融合器處理 prediction / update
- 對每個模組建立 quality score
- 用 guided session 的真實資料做調參

換句話說：

`資料蒐集流程已經開始建立，但正式定位核心必須從 heuristic 轉向 EKF / Error-State KF + handover + quality-driven fusion。`
